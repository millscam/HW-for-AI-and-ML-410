HW/SW partition proposal (sign-language M1 preprocess)

Phase-1 cProfile in project_profile.txt shows the preprocessing frame loop dominates time: handsegment() plus cvtColor(BGR2GRAY), with hot OpenCV primitives, especially bitwise_and, inRange, and VideoCapture.read. We propose hardware acceleration for the fused per-frame segmentation path modeled in ai_calculation.md: dual HSV inRange thresholds, mask bitwise_or, masked bitwise_and on BGR, then fixed-point grayscale. That slice is both time-dominant and analytically volume-heavy (about 0.955 FLOP-equivalent per byte under the documented no-cache-reuse byte accounting).

Roofline rationale: On the plotted AMD Ryzen 9 4900HS target (about 1100 GFLOP/s theoretical FP32 peak, 51.2 GB/s DRAM peak, ridge pi over beta about 21.5 FLOP/byte), the kernel sits left of the ridge at AI about 0.955, so it is in the memory-bound region of the CPU roofline in roofline_project.png. Yet measured throughput is only about 1.7 GFLOP/s, far below the about 49 GFLOP/s memory-ridge ceiling at that AI, so the implementation is also dramatically under the attainable memory roof, which strongly supports a custom datapath (fusion, parallelism, and on-chip reuse) instead of marginal CPU tuning.

Software baseline: The host retains video decode and VideoCapture I/O, Python orchestration, batching and persistence, and later pipeline stages (spatial CNN and RNN in TensorFlow per profiling notes), plus integration, validation, and CPU fallback.

Interface bandwidth: To sustain a 512 GFLOP/s accelerator target without an interface bottleneck at AI about 0.955, the link must deliver roughly 512 divided by 0.955, about 536 GB/s, of effective operand throughput (same FLOP-equivalent convention as the roofline note), an order of magnitude above 51.2 GB/s DRAM peak, implying wide streaming links and/or large local memory close to the accelerator.

Bound shift: On current hardware the kernel is memory-bound in roofline terms (AI much less than the CPU ridge), though far from the memory ridge in practice. The hypothetical accelerator in the plot (512 GFLOP/s peak, about 1800 GB/s on-chip) has a low on-chip ridge (about 0.28 FLOP/byte), so at AI about 0.955 the same workload becomes compute-bound on-chip, provided the host-to-accelerator interface meets the about 536 GB/s requirement above; otherwise the bottleneck simply moves to the interface.
